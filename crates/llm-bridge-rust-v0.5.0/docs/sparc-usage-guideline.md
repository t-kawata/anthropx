## 1. 文档目标

统一团队在使用 SPARC 时的入口选择、协作方式与阶段约束，避免小任务误用集群、大任务误用单 Agent、以及将 `tdd` 错误当作多智能体入口。

目标：提升任务入口选择一致性、复杂任务协作稳定性，以及多智能体闭环成功率。

## 2. 适用范围

本规范适用于日常研发、缺陷修复、测试补齐、代码审查、新功能开发、跨模块重构、安全/性能专项改造、方案设计与技术调研等场景。

- 适用对象：工程师、Tech Lead、项目负责人，以及所有使用 SPARC 或多智能体工作流的团队成员。
- 不解决的问题：本规范不替代具体编码规范、测试规范、安全规范，而是负责解决“任务该从哪个入口进入”。

## 3. 核心原则

- 小任务优先使用单个专家 Agent；复杂任务优先由 `orchestrator` 或 `swarm-coordinator` 接管。
- `TDD` 只作为流程规则与阶段门禁，不作为复杂任务默认入口。
- 高风险任务禁止直接交给单个 `coder` 全包，必须保留测试与审查阶段。
- 集群已存在，不代表任务已进入集群；必须显式选择协调器入口。

## 4. 模式分类

### 入口模式

`orchestrator`、`swarm-coordinator`、`workflow-manager`、`batch-executor` 用于组织任务、控制顺序、汇总结果。

### 专家模式

`coder`、`tester`、`reviewer`、`architect`、`researcher`、`debugger`、`optimizer`、`documenter`、`tdd` 用于解决具体子问题。

## 5. 默认选型规则

- 默认单 Agent：单文件/单模块、目标明确、低风险、无需任务拆解或并行。
- 默认协调器：需要分析 → 实现 → 测试 → 审查，多模块，可能回环修复。
- 默认 Swarm：明确要求多智能体、需要强制分工、需要 delegate 可见性或较强并发能力。

## 6. 场景速查矩阵

| 任务场景 | 推荐入口 | 是否建议集群 | 说明 |
| --- | --- | --- | --- |
| 修一个已定位的小 bug | `coder` | 否 | 范围明确、单能力任务，协调成本通常高于收益。 |
| 给现有逻辑补测试 | `tester` | 否 | 目标是提升覆盖率与验证强度，不需要完整编排。 |
| 做完整功能开发 | `orchestrator` | 是 | 天然包含分析、实现、测试、审查多个阶段。 |
| 严格多智能体 TDD | `orchestrator` / `swarm-coordinator` | 是 | 应将 TDD 作为门禁，由 `tester → coder → reviewer` 推进。 |
| 跨模块重构 | `swarm-coordinator` | 是 | 高风险、高影响面，应显式分工并保留审查。 |
| 技术调研 / 方案对比 | `researcher` | 视规模 | 单问题调研可单 Agent，多方案长期调研可升级协调器。 |

经验法则：明确小任务走专家；跨阶段、跨模块、高风险任务走协调器。

## 7. 任务决策树

- 第一问：只需要一种能力吗？如果答案是“是”，优先单个专家 Agent，例如 `coder`、`tester`、`reviewer`。
- 第二问：会经历多个阶段吗？如果要经历分析、实现、测试、审查等多个阶段，优先使用 `orchestrator`。
- 第三问：是否明确要求多智能体？如果必须强制分工、需要可见 delegate 流程，优先使用 `swarm-coordinator`。
- 第四问：是否要求 TDD？如果需要 TDD，请把它当作规则和门禁，而不是直接当作复杂任务入口。

## 8. TDD 使用规范

统一约定：复杂任务禁止直接以 `tdd` 作为入口；正确方式是 `orchestrator/swarm-coordinator + TDD mandatory`。

- 推荐顺序：`tester → coder → reviewer`
- 最小门禁：`TEST_PLAN_READY`、`RED_CONFIRMED`、`IMPLEMENTATION_DONE`、`REVIEW_CLEAR`
- 未满足 failing tests 前，不得进入实现；未通过 review，不得宣告完成。

## 9. 高风险任务规范

以下任务默认视为高风险：跨模块重构、公共库修改、协议兼容改动、安全逻辑修改、影响多个调用方的接口变更、高回归风险优化。

- 不得直接交给单个 `coder` 全包
- 必须经过 `reviewer`
- 建议经过 `tester`
- 优先使用 `orchestrator` 或 `swarm-coordinator`

## 10. 标准工作流

- 小任务工作流：专家 Agent 直达，修 bug 用 `coder`，补测试用 `tester`，快速审查用 `reviewer`。
- 常规功能工作流：`orchestrator` 开局 → `coder` 实现 → `tester` 验证 → `reviewer` 把关 → 汇总结果。
- 多智能体 TDD 工作流：`orchestrator/swarm-coordinator` 开局 → `tester` → `coder` → `tester` → `reviewer`。

## 11. 常见反模式

- 用 `tdd` 直接启动复杂任务，并期待自动多智能体分工。
- 所有任务默认都先走 swarm。
- 高风险任务直接交给单个 `coder` 从头做到尾。
- 已启用 swarm，但不显式要求协调器接管任务。
- 没有测试/审查门禁，就直接宣告完成。

## 12. 提示词模板

- 小任务：使用 `coder` 处理 [任务]，范围限制在 [模块/文件]，只完成当前目标，不做额外扩展。
- 复杂任务：使用 `orchestrator` 执行 [任务]，请拆解阶段、分配角色、汇总结果，主入口 Agent 不要直接包办全部实现。
- 多智能体 TDD：使用 `orchestrator` 执行 [任务]，按 `tester → coder → reviewer` 顺序推进，没有 failing tests 不得进入实现，没有 review clear 不得结束。

## 13. 启动前检查清单

- 这是小任务还是复杂任务？
- 是否需要多个角色协作？
- 是否需要测试/审查门禁？
- 是否已选择正确入口模式？
- 是否明确禁止主 Agent 越权包办？

## 14. 质量观察指标

| 指标 | 关注点 | 观察目的 |
| --- | --- | --- |
| 单 Agent 任务平均完成时长 | 是否存在小任务过度协同 | 评估入口是否过重 |
| 集群任务平均回环次数 | 测试/审查后回退次数 | 评估复杂任务拆解质量 |
| TDD 任务测试先行比例 | 是否先有 failing tests | 防止名义 TDD、实际直写代码 |
| 高风险任务 review 覆盖率 | 是否经过 reviewer | 降低回归与质量事故 |
| 误用入口导致的返工率 | 是否因入口选错而重做 | 优化团队使用习惯 |

## 15. 团队落地建议

- 第一步：统一认知。先让团队分清“入口模式”和“专家模式”，尤其要明确 `tdd` 不是复杂任务默认入口。
- 第二步：统一提任务方式。要求每个复杂任务显式写清：任务类型、入口模式、是否需要 TDD 门禁、是否要求多智能体分工。
- 第三步：统一复盘标准。复盘任务是否出现主 Agent 一把梭、是否跳过测试/审查、是否存在不必要的集群开销。
- 第四步：挂入知识库。将本规范放在团队文档首页，并与提示词模板库、TDD 规范、高风险任务处理规范互相链接。

## 16. 新成员 5 分钟上手

先记一句：小任务找专家，大任务找协调器；`TDD` 是规则，不是入口。

- 第 1 分钟：先判断任务大小。如果是单文件、单模块、目标明确的小任务，优先用 `coder`、`tester`、`reviewer` 等专家 Agent。
- 第 2 分钟：再判断是否多阶段。如果任务需要分析、实现、测试、审查串起来，默认先用 `orchestrator`。
- 第 3 分钟：是否要求强制协作。如果需要显式分工、delegate 可见性、或明确要求多智能体，改用 `swarm-coordinator`。
- 第 4 分钟：如果要求 TDD。不要直接用 `tdd` 开局，而是写明：`orchestrator + TDD mandatory + tester → coder → reviewer`。
- 第 5 分钟：启动前自检。确认 5 件事：任务大小、入口模式、是否需要门禁、是否要多智能体、是否禁止主 Agent 一把梭。
- 推荐最短模板：复杂任务使用 `orchestrator` 执行 [任务]；按阶段推进，主入口 Agent 不要直接包办全部实现。

## 17. 错误用法 vs 正确用法

| 场景 | 错误用法 | 正确用法 |
| --- | --- | --- |
| 严格 TDD 的复杂任务 | 直接用 `tdd` 启动，并期待自动多智能体分工 | 使用 `orchestrator` 或 `swarm-coordinator` 开局，并声明 `TDD mandatory` |
| 一个已定位的小 bug | 默认先走 swarm，拉起多个角色一起处理 | 直接使用 `coder`，必要时补一个 `tester` 验证即可 |
| 跨模块重构 | 直接交给单个 `coder` 从头做到尾 | 使用 `swarm-coordinator` 或 `orchestrator`，保留测试与审查阶段 |
| 已有 swarm 环境 | 以为“集群已启动”就一定会自动协同 | 显式指定协调器接管任务，并写明是否需要强制分工 |
| 普通复杂功能开发 | 主 Agent 自己分析、自己编码、自己验证、自己宣布完成 | 使用 `orchestrator`，拆解为分析 → 实现 → 测试 → 审查 → 汇总 |
| 补测试场景 | 先找 `coder` 顺手补一点测试 | 直接使用 `tester`，明确覆盖正常路径、边界条件和错误路径 |

团队内复盘时，可以直接用本表判断：本次任务是否“入口选错”或“角色分工过重 / 过轻”。
